9.1 ทำความรู้จัก OpenAI API

OpenAI API เป็น REST API ที่ให้คุณเรียกใช้ความสามารถของโมเดล GPT ผ่านโค้ดได้โดยตรง การยืนยันตัวตนทำผ่าน API Key ที่ส่งใน HTTP Header ทุกครั้งที่เรียก endpoint

1. สมัคร OpenAI

สมัครบัญชีที่ platform.openai.com และเติม Credit เพื่อใช้งาน API

2. รับ API Key

ไปที่ API Keys ในแดชบอร์ด คลิก "Create new secret key" และบันทึกไว้ให้ดี เพราะดูได้แค่ครั้งเดียว

3. ติดตั้ง Library

รันคำสั่ง pip install openai ในเทอร์มินัลของคุณ

4. เริ่มใช้งาน

เรียก client.chat.completions.create() พร้อม model และ messages แล้วรับคำตอบได้เลย

ตัวอย่างการเรียก OpenAI API

import openai

client = openai.OpenAI(api_key="sk-...")

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "อธิบาย Python แบบง่ายๆ"}
    ],
    temperature=0.7,
    max_tokens=500
)
print(response.choices[0].message.content)

พารามิเตอร์สำคัญของ API

  • model — ชื่อโมเดลที่ต้องการใช้ เช่น gpt-4o, gpt-4o-mini
  • messages — อาร์เรย์ของบทสนทนา แต่ละรายการมี role (system / user / assistant) และ content
  • temperature — ควบคุมความสร้างสรรค์ ค่า 0–2; ต่ำ = มีระเบียบ, สูง = สร้างสรรค์กว่า
  • max_tokens — จำนวน Token สูงสุดของคำตอบ ใช้ควบคุมค่าใช้จ่ายและความยาว
  • top_p — Nucleus Sampling อีกวิธีควบคุม Randomness แนะนำให้ใช้แค่ temperature หรือ top_p อย่างใดอย่างหนึ่ง

9.2 Anthropic Claude API

Anthropic Claude API มีโครงสร้างคล้าย OpenAI แต่มีความแตกต่างในการส่ง system prompt และ format ของ response รองรับ Claude 3 / Claude 3.5 / Claude Opus 4 series

OpenAI API Style

  • system อยู่ใน messages array
  • ตอบกลับที่ response.choices[0].message.content
  • ใช้ openai.OpenAI() client
  • parameter: temperature, max_tokens
  • Models: GPT-4o, GPT-4o-mini, o1, o3

Claude API Style

  • system เป็น parameter แยกออกมา
  • ตอบกลับที่ message.content[0].text
  • ใช้ anthropic.Anthropic() client
  • parameter: max_tokens (บังคับ), temperature
  • Models: Claude Opus 4, Sonnet 4, Haiku 3.5

ตัวอย่างการเรียก Claude API

import anthropic

client = anthropic.Anthropic(api_key="sk-ant-...")

message = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    system="คุณเป็นผู้ช่วย AI ที่เชี่ยวชาญด้านวิทยาศาสตร์",
    messages=[
        {"role": "user", "content": "อธิบาย Quantum Computing"}
    ]
)
print(message.content[0].text)

9.3 Function Calling & Tool Use

Function Calling คือความสามารถที่ให้ AI เรียกใช้ฟังก์ชันจริงในโค้ดของคุณได้ แทนที่จะตอบเป็นข้อความเพียงอย่างเดียว โมเดลจะตัดสินใจว่าเมื่อไหร่ควรเรียก function ใดและส่ง argument อะไรไป

1. User

ส่งคำถาม เช่น "อุณหภูมิกรุงเทพวันนี้เท่าไหร่?"

2. LLM วิเคราะห์

โมเดลรู้จัก tools ที่มี ตัดสินใจเรียก get_weather(city="Bangkok")

3. Execute Function

โค้ดของคุณรับ function name + arguments แล้วเรียกฟังก์ชันจริง

4. Return Result

ส่งผลลัพธ์กลับไปให้โมเดลเพื่อประกอบคำตอบสุดท้าย

5. Response

โมเดลสร้างคำตอบภาษาธรรมชาติโดยใช้ข้อมูลจริงจาก function

ตัวอย่าง Function Calling กับ OpenAI

import json

def get_weather(city: str, unit: str = "celsius") -> dict:
    # Mock weather API
    return {"city": city, "temp": 28, "unit": unit, "condition": "Sunny"}

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get current weather for a city",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "City name"},
                    "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
                },
                "required": ["city"]
            }
        }
    }
]

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "อุณหภูมิกรุงเทพวันนี้เท่าไหร่?"}],
    tools=tools,
    tool_choice="auto"
)

# Handle tool call
if response.choices[0].message.tool_calls:
    tool_call = response.choices[0].message.tool_calls[0]
    args = json.loads(tool_call.function.arguments)
    result = get_weather(**args)
    print(f"Weather result: {result}")

Use Cases ของ Function Calling

  • ดึงข้อมูล Real-time: สภาพอากาศ, ราคาหุ้น, ข่าวล่าสุด
  • CRUD Database: สร้าง / อ่าน / แก้ไข / ลบข้อมูลในฐานข้อมูล
  • เรียก External Service: ส่ง Email, สร้าง Calendar Event, โพสต์ Social Media
  • คำนวณและประมวลผลที่ซับซ้อน: คณิตศาสตร์, การแปลงหน่วย, Data Analysis

9.4 Streaming Responses

Streaming คือการรับ response แบบ real-time ทีละ token แทนที่จะรอจนโมเดลสร้างคำตอบครบทั้งหมด ทำให้ผู้ใช้รู้สึกว่าแอปตอบสนองเร็วขึ้นมาก

Chatbot UI

แสดงข้อความทีละตัวอักษรเหมือน ChatGPT ทำให้ UX ดูเป็นธรรมชาติและรู้สึกเร็ว

Long Documents

เมื่อโมเดลต้องสร้างเนื้อหายาว Streaming ช่วยให้ผู้ใช้เริ่มอ่านได้ทันทีโดยไม่ต้องรอนาน

Perceived Performance

Time-to-first-token ลดลงมาก ทำให้แอปดูเร็วกว่าความเป็นจริง

Cancel Mid-stream

ผู้ใช้สามารถหยุดการสร้างคำตอบกลางทางได้ ประหยัด token และ cost

Streaming กับ OpenAI และ FastAPI

# Streaming with OpenAI
stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "เขียนบทกวีเกี่ยวกับ AI"}],
    stream=True
)

for chunk in stream:
    if chunk.choices[0].delta.content is not None:
        print(chunk.choices[0].delta.content, end="", flush=True)

# Streaming with FastAPI
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.get("/chat")
async def chat_stream(message: str):
    async def generate():
        stream = client.chat.completions.create(
            model="gpt-4o",
            messages=[{"role": "user", "content": message}],
            stream=True
        )
        for chunk in stream:
            if chunk.choices[0].delta.content:
                yield f"data: {chunk.choices[0].delta.content}\n\n"
    return StreamingResponse(generate(), media_type="text/event-stream")

ข้อควรระวัง Streaming

  • ไม่สามารถดู usage (จำนวน token) ใน streaming response ได้โดยตรง ต้องนับเอง
  • Error จะเกิดกลางสตรีมได้ ต้องจัดการ exception ใน generator
  • Frontend ต้องรองรับ Server-Sent Events (SSE) หรือ WebSocket

9.5 สร้าง AI Chatbot Application

Chatbot ที่ใช้งานได้จริงต้องจำ บทสนทนาทั้งหมด ไว้ใน conversation history และส่งไปพร้อมกันทุกครั้งที่เรียก API เพราะ LLM ไม่มี memory ระหว่างการเรียก

Chatbot Class พร้อม Conversation History

from openai import OpenAI
import json

class AIChatbot:
    def __init__(self, system_prompt: str, model: str = "gpt-4o"):
        self.client = OpenAI()
        self.model = model
        self.conversation_history = [
            {"role": "system", "content": system_prompt}
        ]

    def chat(self, user_message: str) -> str:
        self.conversation_history.append({
            "role": "user",
            "content": user_message
        })

        response = self.client.chat.completions.create(
            model=self.model,
            messages=self.conversation_history,
            temperature=0.7,
            max_tokens=1000
        )

        assistant_message = response.choices[0].message.content
        self.conversation_history.append({
            "role": "assistant",
            "content": assistant_message
        })

        return assistant_message

    def save_history(self, filename: str):
        with open(filename, 'w', encoding='utf-8') as f:
            json.dump(self.conversation_history, f, ensure_ascii=False, indent=2)

    def get_token_usage(self) -> dict:
        # Estimate tokens (rough calculation)
        total_chars = sum(len(m["content"]) for m in self.conversation_history)
        return {"estimated_tokens": total_chars // 4}

# Usage
bot = AIChatbot(
    system_prompt="คุณเป็นผู้ช่วยสอนวิชาคณิตศาสตร์ อธิบายเป็นภาษาไทย ใช้ตัวอย่างที่เข้าใจง่าย"
)

print(bot.chat("อธิบาย Derivative คืออะไร"))
print(bot.chat("ช่วยยกตัวอย่างการใช้จริงให้หน่อย"))
bot.save_history("conversation.json")

Tips: การจัดการ Conversation History

  • Token Limit Management — เมื่อ history ยาวเกินไป ให้ตัด message เก่าออก หรือสรุปเนื้อหาก่อนหน้า (Summarization)
  • Sliding Window — เก็บเฉพาะ N รอบล่าสุด เช่น self.conversation_history[-20:]
  • Persistent Storage — บันทึก history ลง Database เพื่อให้ผู้ใช้กลับมาคุยต่อในเซสชันใหม่ได้
  • System Prompt — อย่าตัด system message ออกเมื่อ trim history เพราะมันสำคัญมาก

9.6 Error Handling & Rate Limiting

การเรียก API จากภายนอกมีโอกาสเกิด error เสมอ การเขียนโค้ดที่ดีต้องดักจับ error แต่ละประเภทและจัดการอย่างเหมาะสม โดยเฉพาะ Rate Limit ที่พบบ่อยในช่วงเริ่มต้น

RateLimitError

ส่ง request เร็วเกินที่ API กำหนด แก้ด้วย Exponential Backoff — รอ 5s, 10s, 20s…

AuthenticationError

API Key ผิดหรือหมดอายุ ไม่ควร retry แต่ควรแจ้ง user ให้ตรวจสอบ credentials

APIError (5xx)

Server ของ OpenAI มีปัญหา สามารถ retry ได้หลังรอสักครู่

Timeout

Request ใช้เวลานานเกิน limit ที่กำหนด ควรตั้ง timeout=30 เสมอ

Retry Logic ด้วย Exponential Backoff

import time
import openai
from openai import RateLimitError, APIError, AuthenticationError

def call_api_with_retry(messages: list, max_retries: int = 3) -> str:
    client = openai.OpenAI()

    for attempt in range(max_retries):
        try:
            response = client.chat.completions.create(
                model="gpt-4o",
                messages=messages,
                timeout=30
            )
            return response.choices[0].message.content

        except RateLimitError:
            wait_time = (2 ** attempt) * 5  # Exponential backoff
            print(f"Rate limit hit. Waiting {wait_time}s...")
            time.sleep(wait_time)

        except AuthenticationError:
            raise ValueError("Invalid API Key - check your credentials")

        except APIError as e:
            print(f"API Error: {e}. Retrying...")
            time.sleep(2)

    raise Exception(f"Failed after {max_retries} attempts")

Best Practices: Security & Reliability

ใช้ Environment Variables

เก็บ API Key ใน .env file หรือ Secret Manager ไม่ใช่ใน source code

ไม่ Hardcode API Key

ไม่ commit API Key ลง Git เด็ดขาด ถ้าเผลอ commit ให้ revoke ทันที

Implement Retry

ทุก production code ควรมี retry logic สำหรับ RateLimitError และ APIError

Set Timeout

กำหนด timeout เสมอ เพื่อป้องกัน request ค้างนานจนแอปหยุดตอบสนอง

9.7 Cost Optimization & Deployment

ค่าใช้จ่ายของ API คิดตาม Token — ทั้ง input และ output รวมกัน การเลือกโมเดลและจัดการ prompt ดีๆ ช่วยลดต้นทุนได้มาก

ราคา Token ของโมเดลยอดนิยม (ราคาโดยประมาณ ปี 2025)

GPT-4o-mini $0.15 $0.60 128K งานเบา, ประหยัดสุด

เทคนิคลดค่าใช้จ่าย

Caching

Cache คำตอบสำหรับ prompt ที่ซ้ำกัน ด้วย Redis หรือ in-memory dict ลด API call ซ้ำซ้อน

Prompt Optimization

เขียน prompt กระชับ ไม่ใส่ข้อมูลที่ไม่จำเป็น ลด input token โดยตรง

เลือกโมเดลให้เหมาะ

ใช้ GPT-4o-mini หรือ Claude Haiku สำหรับงานง่าย ประหยัดได้ถึง 90% เทียบกับ flagship

Batch Processing

รวม request หลายอันให้เป็น batch เดียวเมื่อเป็นไปได้ ลดจำนวนการเรียก API

การ Deploy: ตัวเลือกยอดนิยม

Cloud Platforms

  • Render — Free tier มี, เหมาะสำหรับ prototype
  • Heroku — Classic, ง่ายแต่ราคาสูงกว่า

9.8 สรุปและแบบฝึกหัด

สรุปสิ่งที่เรียนในบทนี้

  • OpenAI API — ใช้ client.chat.completions.create() พร้อม API Key สำหรับ Authentication
  • Claude API — โครงสร้างคล้ายกัน แต่ system prompt แยกเป็น parameter และตอบกลับที่ message.content[0].text
  • Function Calling — ให้ AI ตัดสินใจเรียกฟังก์ชันในโค้ดของคุณเพื่อดึงข้อมูล Real-time หรือทำ Action ต่างๆ
  • Streaming — รับ response ทีละ token ด้วย stream=True เหมาะกับ chatbot UI และเนื้อหายาว
  • Conversation History — ต้องส่ง history ทั้งหมดทุกครั้งที่เรียก API เพราะ LLM ไม่มี memory ในตัว
  • Error Handling — ดักจับ RateLimitError, AuthenticationError, APIError และ implement Exponential Backoff
  • Cost Optimization — เลือกโมเดลให้เหมาะ, ใช้ caching, optimize prompt และ set usage limits
  • Security — เก็บ API Key ใน environment variables เสมอ ห้าม hardcode ใน source code
