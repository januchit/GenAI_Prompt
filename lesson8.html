8.1 ปัญหาของ LLM และทางแก้

LLM มีข้อจำกัดสำคัญที่ทำให้ไม่สามารถใช้งานได้ทุกสถานการณ์ โดยเฉพาะในองค์กรที่ต้องการคำตอบจากข้อมูลภายใน

Knowledge Cutoff

LLM ถูกฝึกถึงวันที่หนึ่ง ไม่รู้ข้อมูลหลังจากนั้น เช่น GPT-4 cutoff ต้นปี 2024

Private Data

LLM ไม่รู้ข้อมูลภายในองค์กร เช่น คู่มือ, นโยบาย, ข้อมูลลูกค้า, รายงานภายใน

Hallucination

เมื่อ LLM ไม่มีข้อมูล มักสร้างคำตอบที่ฟังดูถูกต้องแต่ผิดข้อเท็จจริง

No Source Citation

ไม่สามารถอ้างอิงแหล่งที่มาได้ ทำให้ตรวจสอบความถูกต้องได้ยาก

RAG คือทางแก้!

แทนที่จะ Fine-tune LLM ด้วยข้อมูลใหม่ (แพงและช้า) RAG ให้ LLM ค้นหาข้อมูลที่เกี่ยวข้องก่อน แล้วนำมาประกอบคำตอบ ได้ข้อมูลล่าสุด อ้างอิงได้ และลด Hallucination

8.2 RAG คืออะไร?

Retrieval-Augmented Generation (RAG) คือสถาปัตยกรรมที่ผสานระหว่าง การค้นหาข้อมูล (Retrieval) กับ การสร้างคำตอบ (Generation) ทำให้ LLM ตอบคำถามจากข้อมูลของคุณเองได้

RAG vs Fine-tuning

RAG

  • อัปเดตข้อมูลได้ง่าย
  • อ้างอิงแหล่งข้อมูลได้
  • ราคาถูกกว่ามาก
  • เริ่มใช้งานได้เร็ว

Fine-tuning

  • AI เรียนรู้ Style ของข้อมูล
  • ตอบสนองเร็วกว่าใน Inference
  • เหมาะกับ Task เฉพาะทาง
  • ราคาแพง ใช้เวลานาน

Chunking Strategy

ก่อน Embed ต้องแบ่งเอกสารเป็น Chunk เล็กๆ การเลือก Chunk Size ส่งผลต่อคุณภาพการค้นหามาก

Chunk ใหญ่เกิน (1000+ tokens)

Context เยอะ แต่ค้นหาไม่แม่นยำ ดึงข้อมูลที่ไม่เกี่ยวข้องมาด้วย

Chunk พอดี (256–512 tokens) ✅

สมดุลระหว่าง Context และความแม่นยำ เหมาะกับงานส่วนใหญ่

Chunk เล็กเกิน (50 tokens)

ค้นหาแม่นยำ แต่ Context น้อยเกินไป AI ไม่มีข้อมูลพอตอบ

8.5 RAG Pipeline ทีละขั้นตอน

1

Document Loading

โหลดเอกสารจากแหล่งต่างๆ: PDF, Word, Excel, Web Pages, YouTube, GitHub

from langchain_community.document_loaders import PyPDFLoader

loader = PyPDFLoader("company_handbook.pdf")
documents = loader.load()  # โหลดทุกหน้า
2

Text Splitting

แบ่งเอกสารเป็น Chunk เล็กๆ พร้อม Overlap เพื่อไม่ให้ขาดบริบท

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,      # ขนาด Chunk
    chunk_overlap=50,   # Overlap ระหว่าง Chunk
)
chunks = splitter.split_documents(documents)
3

Embedding & Storing

แปลง Chunk เป็น Embedding Vector แล้วเก็บใน Vector Database

from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)
4

Retrieval

เมื่อมีคำถาม ค้นหา Chunk ที่ใกล้เคียงที่สุดใน Vector DB

retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 5}  # ดึง Top-5 Chunks
)

query = "นโยบายการลาป่วยมีกี่วัน?"
relevant_docs = retriever.invoke(query)
5

Generation

ส่ง Context ที่ค้นมา + คำถาม ไปยัง LLM เพื่อสร้างคำตอบ

from langchain_openai import ChatOpenAI
from langchain.chains import RetrievalQA

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True  # อ้างอิงแหล่งที่มา
)

result = qa_chain.invoke({"query": query})
print(result["result"])
print(result["source_documents"])  # แหล่งอ้างอิง

8.6 สร้าง RAG ด้วย LangChain (Full Example)

Python — RAG System Completerag_system.py
from langchain_community.document_loaders import PyPDFLoader, WebBaseLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_chroma import Chroma
from langchain.chains import ConversationalRetrievalChain
from langchain.memory import ConversationBufferMemory

class RAGSystem:
    """RAG System ที่รองรับการสนทนาต่อเนื่อง"""

    def __init__(self, pdf_paths: list[str]):
        # 1. Load Documents
        docs = []
        for path in pdf_paths:
            loader = PyPDFLoader(path)
            docs.extend(loader.load())

        # 2. Split
        splitter = RecursiveCharacterTextSplitter(
            chunk_size=512, chunk_overlap=64
        )
        chunks = splitter.split_documents(docs)

        # 3. Embed & Store
        embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
        self.vectorstore = Chroma.from_documents(chunks, embeddings)

        # 4. Setup Chain with Memory
        self.memory = ConversationBufferMemory(
            memory_key="chat_history",
            return_messages=True
        )
        self.chain = ConversationalRetrievalChain.from_llm(
            llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),
            retriever=self.vectorstore.as_retriever(
                search_kwargs={"k": 4}
            ),
            memory=self.memory,
            return_source_documents=True
        )

    def ask(self, question: str) -> dict:
        result = self.chain.invoke({"question": question})
        sources = list({
            doc.metadata.get("source", "Unknown")
            for doc in result["source_documents"]
        })
        return {
            "answer": result["answer"],
            "sources": sources
        }

# การใช้งาน
rag = RAGSystem(["company_policy.pdf", "employee_handbook.pdf"])
result = rag.ask("ลาพักร้อนได้กี่วันต่อปี?")
print(f"คำตอบ: {result['answer']}")
print(f"แหล่งอ้างอิง: {result['sources']}")

8.7 Advanced RAG Techniques

Hybrid Search

ผสม Semantic Search (Vector) กับ Keyword Search (BM25) ได้ผลดีกว่าใช้อย่างเดียว เหมาะกับภาษาไทยที่มีคำเฉพาะ

Reranking

หลังดึง Top-K Chunks มา ใช้ Reranker Model (เช่น Cohere Rerank) เรียงลำดับใหม่ให้ตรงกับคำถามมากขึ้น

Query Expansion

ให้ LLM สร้างคำถามหลายรูปแบบจากคำถามเดิม แล้ว Search ทุกรูปแบบ รวม Result ด้วย Reciprocal Rank Fusion

Parent Document Retriever

Embed Chunk เล็ก (แม่นยำ) แต่ส่ง Parent Chunk ใหญ่กลับ (Context ครบ) ได้ทั้งความแม่นยำและบริบท

RAPTOR

สร้าง Summary Tree แบบ Hierarchical Cluster ช่วยตอบคำถามที่ต้องการ High-level Overview

Self-RAG

LLM ตัดสินใจเองว่าต้อง Retrieve ไหม ตรวจสอบคุณภาพ Retrieval และ Generation ด้วยตัวเอง

RAG Evaluation

Context Precision

ใน Context ที่ดึงมา มีส่วนที่เกี่ยวข้องกับคำถามมากแค่ไหน

Context Recall

ข้อมูลที่จำเป็นสำหรับตอบคำถามถูกดึงมาครบไหม

Faithfulness

คำตอบที่ LLM สร้างขึ้นตรงกับ Context ที่ให้ไปหรือไม่ (ไม่ Hallucinate)

Answer Relevancy

คำตอบตอบตรงคำถามหรือไม่ ไม่ออกนอกเรื่อง

Framework สำหรับ Evaluate RAG
RAGAS — ประเมิน RAG Pipeline อัตโนมัติด้วย LLM-as-Judge | TruLens — Observability สำหรับ LLM App

8.8 สรุปและแบบฝึกหัด

สรุปสาระสำคัญ

  • RAG แก้ปัญหา Knowledge Cutoff, Private Data, Hallucination และ No Citation ของ LLM
  • RAG Pipeline: Load → Split → Embed → Store → Retrieve → Generate
  • Embedding แปลงข้อความเป็น Vector เพื่อวัด Semantic Similarity
  • Chunk Size 256–512 tokens มักได้ผลดีที่สุด ต้องทดลองสำหรับแต่ละ Domain
  • Vector DB หลัก: Chroma (เริ่มต้น), Pinecone (Cloud), Qdrant (Performance), pgvector (PostgreSQL)
  • Advanced: Hybrid Search, Reranking, Query Expansion ช่วยเพิ่มคุณภาพสูงมาก

แบบฝึกหัดท้ายบท

1
ทดลอง Semantic Search

ใช้ OpenAI Embedding API แปลงประโยค 10 ประโยคเป็น Vector แล้วคำนวณ Cosine Similarity ระหว่างประโยค วาด Scatter Plot แสดงความสัมพันธ์

2
สร้าง RAG อย่างง่าย

สร้าง RAG System สำหรับเอกสาร PDF 1 ไฟล์ (เช่น คู่มือวิชา) ด้วย LangChain + Chroma ทดสอบด้วยคำถามอย่างน้อย 5 ข้อ

3
เปรียบเทียบ Chunk Size

ทดสอบ RAG ด้วย Chunk Size ต่างกัน (128, 256, 512, 1024) บันทึกคุณภาพคำตอบและวิเคราะห์ว่า Chunk Size ใดเหมาะกับเอกสารของคุณ

4
Mini Project: Knowledge Base Chatbot

สร้าง Chatbot ที่ตอบคำถามจากเอกสารของมหาวิทยาลัยหรือองค์กรที่สนใจ ใส่ Feature อ้างอิงแหล่งที่มา และ Conversation Memory
